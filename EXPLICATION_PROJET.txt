 C'EST QUOI CE PROJET ?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

RAG = Retrieval-Augmented Generation.

Principe : au lieu de demander directement à un LLM (qui ne connaît pas tes
documents internes), on fait deux choses :

  1. On cherche dans une base vectorielle les morceaux de texte pertinents par
     rapport à la question.
  2. On les envoie au LLM comme "contexte" pour qu'il génère une réponse
     basée UNIQUEMENT sur ces documents.

Résultat : un chatbot local qui répond à partir de tes PDFs, tourne 100% en
local (aucune donnée ne part sur internet), et dit "je ne sais pas" si la
réponse n'est pas dans les documents.

 C'EST QUOI LA SIMILARITY SEARCH ?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

La recherche par similarité (search_type="similarity") est l'approche de base.

Comment ça marche :

  Question → nomic-embed-text → vecteur_question (768 dimensions)

  Pour chaque chunk stocké dans ChromaDB :
    chunk_i → vecteur_chunk_i (768 dimensions)

  On calcule la similarité cosinus entre vecteur_question et chaque vecteur :

    similarité = cos(θ) = (A · B) / (||A|| × ||B||)

    cos = 1   → vecteurs identiques (même sens)
    cos = 0   → vecteurs orthogonaux (sujets sans rapport)
    cos = -1  → vecteurs opposés

  Les K chunks avec le cosinus le plus élevé sont retournés.

Avantage : rapide, simple, efficace pour trouver les passages les plus proches
           sémantiquement de la question.

Inconvénient : si plusieurs chunks très similaires parlent du même sujet, ils
               vont tous sortir ensemble (redondance). Le LLM reçoit alors
               du contexte répétitif au lieu d'informations variées.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 C'EST QUOI MMR (Maximal Marginal Relevance) ?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

MMR est une stratégie de recherche qui équilibre PERTINENCE et DIVERSITÉ.

Formulé par Carbonell & Goldstein (1998), l'idée :
→ ne pas retourner que les chunks les plus proches si ils se ressemblent trop
→ choisir des chunks qui sont pertinents ET différents des déjà sélectionnés

Algorithme MMR :

  1. On commence avec la liste de tous les candidats (fetch_k chunks).
     Dans notre config : fetch_k = RETRIEVAL_K_FETCH * 2 = 20 chunks.

  2. On sélectionne le chunk le plus similaire à la query. → chunk #1 retenu.

  3. Pour chaque chunk restant, on calcule un score MMR :

     MMR(chunk_i) = λ × sim(query, chunk_i)
                  - (1 - λ) × max[ sim(chunk_i, chunk_déjà_retenu) ]

     où sim() est la similarité cosinus,
     et λ est un paramètre d'équilibre (par défaut 0.5 dans LangChain).

     → Le premier terme récompense la pertinence par rapport à la query.
     → Le second terme pénalise les chunks trop similaires à ceux déjà retenus.

  4. On retient le chunk avec le meilleur score MMR. → chunk #2 retenu.

  5. On répète jusqu'à avoir K chunks.

Exemple concret :

  Tu poses la question "Comment déployer le projet ?"
  Les 20 candidats incluent 8 chunks très similaires sur le déploiement.

  - Similarity pure : retourne les 2 chunks les plus proches, tous les deux
    sur le déploiement, quasiment identiques → contexte redondant.

  - MMR : retourne 1 chunk sur le déploiement + 1 chunk sur un aspect connexe
    (ex: prérequis ou configuration) → contexte plus riche.

Dans notre code (retrieval_strategies.py) :

  store.as_retriever(
      search_type="mmr",
      search_kwargs={"k": k_fetch, "fetch_k": k_fetch * 2}
  )

  k       = nombre de chunks à retourner (RETRIEVAL_K_FETCH = 10)
  fetch_k = pool de candidats initial (ici 20)

Quand utiliser MMR vs Similarity ?

  → SIMILARITY : quand la question est précise et ciblée, que tu veux les
    passages les plus proches sans te soucier de la diversité.

  → MMR : quand les documents ont des parties redondantes (slides, rapports
    avec répétitions), ou quand tu veux avoir un contexte plus varié pour
    les questions ouvertes.

 C'EST QUOI LE RERANK BM25 ?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Le rerank est une deuxième passe de scoring, après la recherche vectorielle.

Problème que ça résout :
  La recherche vectorielle est bonne pour le sens général (sémantique), mais
  elle peut louper des correspondances exactes de mots-clés.

  Exemple : tu cherches "CVE-2024-1337". Les embeddings peuvent trouver des
  chunks sur la "cybersécurité" en général, mais rater le chunk qui contient
  littéralement "CVE-2024-1337". BM25 lui, cherche l'exact string.

Approche Hybrid = vectoriel + BM25 (ce qu'on fait ici).

Comment fonctionne BM25 Okapi (utils/rerank.py) :

  BM25 = Best Match 25, variante de TF-IDF avec normalisation par longueur.

  Formule simplifiée pour un terme t dans un document d :

    score(t, d) = IDF(t) × [ TF(t,d) × (k1 + 1) ]
                           ────────────────────────────
                           [ TF(t,d) + k1 × (1 - b + b × |d|/avgdl) ]

  où :
    TF(t,d)  = fréquence du terme t dans le document d
    IDF(t)   = inverse document frequency (rare = plus important)
    |d|      = longueur du document d (en tokens)
    avgdl    = longueur moyenne des documents
    k1, b    = paramètres de saturation (défauts : k1=1.5, b=0.75)

  Le score final d'un document = somme des scores pour tous les termes de la query.

Ce qu'on fait dans notre code :

  1. On prend les 10 chunks retournés par la recherche vectorielle.
  2. On tokenise chaque chunk (lower + split).
  3. On instancie BM25Okapi avec ces 10 chunks comme corpus.
  4. On calcule le score BM25 pour chaque chunk vis-à-vis de la query.
  5. On trie par score décroissant.
  6. On retourne les 2 meilleurs chunks (RETRIEVAL_K = 2).

Attention : ce BM25 est local au mini-corpus des 10 chunks récupérés,
pas sur l'ensemble de la base vectorielle. C'est un reranker léger.

Le pipeline complet avec les chiffres de config :

  ChromaDB → 10 chunks (vectoriel)
           → BM25 rerank
           → 2 chunks finals (envoyés au LLM)

  RETRIEVAL_K       = 2    → Nombre de chunks envoyés au LLM (après rerank)

 FLUX COMPLET QUAND L'UTILISATEUR POSE UNE QUESTION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Utilisateur tape "Comment fonctionne le rapport_2024 ?"
         │
         ▼
  app.py → ask_agent(query, stream=True)
         │
         ▼
  retrieve(query) dans core/retrieve.py
    │
    ├── detect_project_in_query("rapport_2024", [...stems...])
    │     → match trouvé : "rapport_2024"
    │     → filter = {"source_stem": {"$in": ["rapport_2024"]}}
    │
    ├── build_retriever(store, SIMILARITY, k=10, filter=...)
    │     → retriever ChromaDB en mode similarité cosinus
    │
    ├── retriever.invoke("Comment fonctionne le rapport_2024 ?")
    │     → nomic-embed-text encode la query en vecteur
    │     → ChromaDB cherche les 10 chunks les plus proches
    │       (uniquement dans les chunks de rapport_2024)
    │     → retourne 10 Documents
    │
    └── rerank(query, 10 docs)
          → BM25Okapi score chaque chunk sur les mots de la query
          → trie par score décroissant
          → retourne les 2 meilleurs
         │
         ▼
  context = chunk1.page_content + "\n\n" + chunk2.page_content
         │
         ▼
  generate_stream(context, query) dans core/generate.py
    │
    ├── prompt = [SYSTEM: "réponds uniquement depuis le contexte" + context]
    │            [HUMAN: "Comment fonctionne le rapport_2024 ?"]
    │
    └── llama3.1.stream(prompt)
          → tokens envoyés un par un à Streamlit
         │
         ▼
  app.py affiche la réponse progressivement dans l'UI

 DÉTAILS DES UTILITAIRES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

utils/logger.py
  Logger Python standard (module logging) configuré pour sortir sur stdout.
  Niveau DEBUG → tout est loggé (timers, scores BM25, chunks récupérés...).
  Format : "2026-04-30 10:00:00 [DEBUG] core.retrieve — ..."

utils/timer.py
  Context manager qui mesure et logge le temps d'exécution d'un bloc.
  Exemple : "chargement PDF — 1.234s", "retrieve [similarity] — 0.456s"
  Utile pour identifier les goulots d'étranglement.

utils/pdf_to_txt.py
  Lors de l'ingestion, génère un fichier .txt à côté de chaque PDF.
  Chaque page est séparée par "===== SLIDE N =====".
  Sert uniquement à inspecter visuellement le texte extrait par PyPDFLoader.
  N'est pas utilisé dans le pipeline de retrieval.

utils/project_filter.py
  get_source_stem() : extrait le nom de fichier sans extension depuis un path.
  detect_project_in_query() : heuristique de détection de projet dans la query.
    - Filtre les stop words (liste française + anglaise codée en dur)
    - Garde les mots de 4+ caractères
    - Trie par longueur décroissante (les mots longs sont plus discriminants)
    - Cherche si un mot de la query est sous-chaîne d'un stem connu

 COMPARAISON SIMILARITY vs MMR vs RERANK
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Méthode     | Critère principal | Diversité | Exactitude mots-clés | Vitesse
  ────────────|───────────────────|───────────|──────────────────────|────────
  Similarity  | sens sémantique   | non       | faible               | rapide
  MMR         | sens + diversité  | oui       | faible               | moyen
  BM25 seul   | mots exacts       | non       | forte                | rapide
  Hybrid (✓)  | sens + exactitude | selon MMR | forte                | moyen

  Notre pipeline utilise : Similarity (ou MMR) + BM25 rerank = hybrid.

 POUR CHANGER LA STRATÉGIE DE RECHERCHE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Dans config.py, modifier :

    RETRIEVAL_TYPE = RetrievalType.SIMILARITY  # ← actuel
    RETRIEVAL_TYPE = RetrievalType.MMR         # ← passer en MMR

  Pour désactiver le rerank BM25 :
    RERANK_ENABLED = False

  Pour ajuster le nombre de chunks retournés au LLM :
    RETRIEVAL_K = 2  # ← augmenter donne plus de contexte mais ralentit le LLM
